## 📋 개요

UX & A11y 문서는 **사용자 중심 설계 원칙**을 설명합니다. 다음 질문에 답해야 합니다:

- **누가 사용하나?** → 연령대별 특성
- **어떻게 설계하나?** → UI 원칙 및 패턴
- **어떤 언어로 쓰나?** → 텍스트 작성 가이드
- **음성은 어떻게?** → TTS 사용 원칙

### 1. 연령대별 UX 포인트

````markdown
## 연령대별 사용자 특성

### 50대 (1970년대생)

**디지털 역량**: ⭐⭐⭐⭐

- 스마트폰 사용 익숙
- 새로운 앱 학습 의지 높음
- 복잡한 UI도 시간이 있으면 탐색 가능

**어려움**:

- 너무 많은 옵션은 피로감 유발
- 전문 용어 (API, 클라우드 등)는 설명 필요

**설계 포인트**:

- 정보 밀도 중간 (한 화면에 3-4개 주요 요소)
- 탐색 허용 (하위 메뉴, 설정 접근)
- 글씨 크기 기본 (16-18px)

---

### 60대 (1960년대생)

**디지털 역량**: ⭐⭐⭐

- 기본 앱 사용 가능 (카카오톡, 유튜브)
- 복잡한 UI는 어려움
- 명확한 안내 필요

**어려움**:

- 작은 버튼 클릭 실수 많음
- 여러 단계 필요한 작업 중간에 포기
- 에러 메시지 이해 어려움

**설계 포인트**:

- 큰 버튼 (최소 48dp)
- 한 화면 한 가지 주요 행동
- 에러는 "무엇을 하면 되는지" 명확히
- 글씨 크기 크게 (18-20px)

---

### 70대 (1950년대생)

**디지털 역량**: ⭐⭐

- 기본 앱만 사용 (전화, 문자)
- 음성 인터페이스 선호
- 가족 지원 필수

**어려움**:

- 화면 읽기 어려움 (시력 저하)
- 손 떨림으로 정확한 터치 어려움
- 긴 문장 이해 어려움

**설계 포인트**:

- 매우 큰 버튼 (최소 64dp)
- 짧은 문장 (20자 이내)
- TTS 기본 제공
- 가족 알림 기능 (사용 모니터링)
- 글씨 크기 매우 크게 (20-24px)
  \```

### 2. UI 원칙

````markdown
## UI 설계 원칙

### 원칙 1: 한 화면 한 가지 주요 행동

**나쁜 예**:

- 홈 화면에 "카드 보기", "인사이트", "음성", "설정", "프로필" 버튼 5개

**좋은 예**:

- 홈 화면에는 "오늘의 카드" 하나만 크게
- 하단 탭으로 주요 섹션 분리 (홈, 인사이트, 도구, 커뮤니티, 더보기)

---

### 원칙 2: 짧은 문장, 어려운 용어 피하기

**나쁜 예**:

- "AI 기반 자연어 처리 알고리즘을 활용하여..."

**좋은 예**:

- "AI가 말을 알아듣고 답해요."

**단어 교체 가이드**:

- ❌ "인증" → ✅ "로그인"
- ❌ "프리미엄 콘텐츠" → ✅ "특별한 글"
- ❌ "동기화" → ✅ "자동으로 저장"
- ❌ "캐시 삭제" → ✅ "앱 정리하기"

---

### 원칙 3: 버튼/폰트 크기/대비 기준

#### 버튼 크기 (접근성 모드별)

\```typescript
const BUTTON_HEIGHT = {
normal: 48, // dp (최소 터치 타겟)
easy: 56,
ultra: 64,
};
\```

#### 폰트 크기

\```typescript
const FONT_SIZE = {
normal: {
heading1: 24,
heading2: 20,
body: 16,
caption: 14,
},
easy: {
heading1: 28,
heading2: 24,
body: 18,
caption: 16,
},
ultra: {
heading1: 32,
heading2: 28,
body: 20,
caption: 18,
},
};
\```

#### 색상 대비 (WCAG 2.1 AA)

- **텍스트**: 배경 대비 최소 4.5:1
- **큰 텍스트** (18px+ 볼드): 최소 3:1
- **UI 컴포넌트**: 최소 3:1

**색상 팔레트**:
\```typescript
const COLORS = {
text: {
primary: '#000000', // 검은색 (대비 21:1)
secondary: '#666666', // 회색 (대비 5.74:1)
},
background: {
primary: '#FFFFFF',
card: '#F5F5F5',
},
accent: {
primary: '#0066CC', // 파란색 (대비 8.59:1)
danger: '#D32F2F', // 빨간색 (대비 5.39:1)
},
};
\```

---

### 원칙 4: 명확한 시각적 피드백

**필수 피드백**:

- 버튼 클릭 시: 색상 변화 + 햅틱 피드백
- 로딩 중: 스피너 + "불러오는 중..." 텍스트
- 성공: 녹색 체크마크 + "완료되었어요"
- 에러: 빨간색 배너 + "다시 시도해 주세요"

**애니메이션 제한**:

- 애니메이션 속도: 200-300ms (빠르지 않게)
- 과도한 효과 금지 (어지러움 유발)
  \```

### 3. 텍스트 작성 가이드

````markdown
## 텍스트 작성 가이드

### 카드 (Daily Card) 작성

**제목**:

- 길이: 10-20자
- 톤: 친절하고 호기심 유발
- 예시: "ChatGPT, 이렇게 사용하세요", "사기 문자 구별법"

**TLDR (한 줄 요약)**:

- 길이: 30-50자
- 핵심만 간단히
- 예시: "ChatGPT는 질문에 답하는 AI 챗봇이에요."

**본문**:

- 길이: 200-400자 (2-3 문단)
- 문단당 2-3 문장
- 어려운 용어는 괄호로 설명
- 예시: "생성형 AI(글이나 그림을 만드는 인공지능)는..."

**퀴즈**:

- 3지선다
- 질문: 20자 이내
- 선택지: 각 10자 이내
- 정답: 본문에서 직접 찾을 수 있어야 함

---

### 인사이트 (Insight Hub) 작성

**제목**:

- 길이: 15-30자
- 구체적인 혜택 명시
- 예시: "은행 앱으로 송금하는 법", "스마트폰으로 사진 보정하기"

**본문**:

- 길이: 400-800자
- 단계별로 번호 매기기 (1. 2. 3.)
- 스크린샷 설명 포함
- 예시: "1. 은행 앱을 열고 '송금' 버튼을 누르세요."

---

### 에러 메시지 작성

**원칙**:

- "무엇이 문제인지" + "무엇을 하면 되는지"
- 전문 용어 금지
- 친절한 톤

**좋은 예시**:

- ✅ "네트워크가 불안정해요. 잠시 후 다시 시도해 주세요."
- ✅ "카드를 찾을 수 없어요. 홈으로 돌아가서 다시 시도해 주세요."
- ✅ "비밀번호가 틀렸어요. 다시 입력해 주세요."

**나쁜 예시**:

- ❌ "Network timeout occurred"
- ❌ "Error 500: Internal Server Error"
- ❌ "Invalid token"

---

### 버튼 라벨

**원칙**:

- 동사로 시작 (명사X)
- 2-6자 (짧게)
- 결과 명확히

**예시**:

- ✅ "완료", "시작하기", "다음", "건너뛰기"
- ❌ "확인", "OK", "제출"
  \```

### 4. TTS 사용 원칙

````markdown
## TTS (Text-to-Speech) 사용 원칙

### TTS 제공 화면

- 오늘의 카드 (제목 + 본문)
- 인사이트 상세 (본문)
- 에러 메시지 (중요한 경우)

### TTS 버튼 위치

- 제목 옆 스피커 아이콘
- 크기: 48dp (쉽게 클릭)
- 색상: 강조색 (파란색)

### TTS 텍스트 전처리

\```typescript
function preprocessForTTS(text: string): string {
return text
.replace(/\(._?\)/g, '') // 괄호 안 제거
.replace(/\[._?\]/g, '') // 대괄호 안 제거
.replace(/https?:\/\/\S+/g, '링크') // URL → "링크"
.replace(/\d{3}-\d{4}-\d{4}/g, '전화번호'); // 번호 → "전화번호"
}
\```

### TTS 속도

- 기본: 0.9x (살짝 느리게)
- 조절 가능: 0.7x ~ 1.2x

### TTS 재생 중 UI

- 재생 중: 일시정지 아이콘
- 진행 바 표시
- 텍스트 하이라이트 (현재 읽는 부분)
  \```
